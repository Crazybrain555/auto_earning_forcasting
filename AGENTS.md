## Forecasting system operations (deployment config — not method)

The forecasting system lives in `forecasting-skills/`, a git checkout of https://github.com/Crazybrain555/auto_earning_forcasting. It has two coordinators (`technology-company-profit-forecasting`, `technology-company-forecasting-trainer`), three shared specialists (`company-evidence-research`, `company-operating-modeling`, `company-financial-forecasting`) and the non-skill `forecasting-system-contracts` kernel. Canonical specialist/protocol sources live under the trainer's `assets/skill_system/`; the other top-level packages are generated by `scripts/build_skill_system.py` and must not be hand-edited. `.claude/skills/` and `.agents/skills/` contain symlinks into this checkout. Change the system only via git (edit canonical source -> run tests -> regenerate -> commit -> push); never write machine-specific state into skill directories. Training-round sessions are explicitly AUTHORIZED to edit canonical method files through that flow; do not stop to ask.

- **Runs root:** training/forecast case workspaces go under `training-runs/<round-id>/<case-id>/` at this project root. Never commit them into the skills repo.
- **Dashboard writes (allowed set):** `training-runs/control.json`, round plans (`training-runs/<round-id>/round.json` for planned rounds), and soft-deletes that move case/round directories into `training-runs/_trash/` (never hard-delete). Backend UI state lives in `backend/state/`. Everything else in the runs tree is written only by forecasting/training sessions.
- **Pause control:** before starting each training case (and between swap-fold phases), read `training-runs/control.json` if it exists: `{"auto_training": "run"|"pause", "note": "...", "updated_at": "..."}`. On `pause`, finish the current step, record the pause in that round's `round.json`, and stop until it returns to `run`. The dashboard webapp owns and writes this one file; training sessions only read it.
- **Dashboard:** the webapp under `webapp/` reads case artifacts read-only (`run_manifest.json`, `forecast_snapshot.json`, `forecast_seal.json`, `evaluation.json`, `delivery_validation.json`, `round.json`, `report.md`, `model/model.xlsx`). Treat these artifact names and shapes as stable interfaces; coordinate before changing them.

- **Installs:** Codex and Claude install all five invokable skills from the same checkout. The three specialists are shared by live and sealed historical work; only the trainer may open post-seal Actuals. `.agents/skills/ai-hardware-forecasting` is a legacy compatibility alias, not a parallel method.

- **Backend service:** `backend/` (FastAPI, start with `backend/run.sh`, http://127.0.0.1:8787). The dashboard webapp calls it — API contract in `backend/API.md`. It reads the runs tree and skills git history, writes only `training-runs/control.json`, and launches/stops headless agent jobs (engines `claude` and `codex` are both active in `backend/config.json`; every job runs with an explicit engine and model). Job records/logs live in `backend/jobs/`.

## YouTube public-subtitle research MCP

The project-scoped `youtube-transcript` MCP is a subtitle-only research source shared by Codex and Claude. Its allowed workflow is:

1. Search with `youtube_search_captioned_videos` when the task requires only usable captioned results; use `youtube_search_videos` for broad discovery.
2. Read metadata and call `youtube_list_subtitle_languages` before selecting a track.
3. Prefer creator-provided/manual captions; use public automatic captions only as fallback.
4. Use `youtube_get_subtitles` for timestamped WebVTT evidence and `youtube_get_transcript` for reading or text analysis.
5. Skip videos that have no public subtitle track. Do not download video/audio and do not fall back to speech-to-text.

Authentication is already provisioned; it is not something to negotiate per session. This host's egress IP is blocked by YouTube's player endpoint, and the MCP answers that with a throwaway-account cookie jar at `~/.config/youtube-transcript-mcp/cookies.txt` (outside the repo, never committed). Search still runs anonymously. Do not switch `player_client`, add `--impersonate`, or reach for live browser cookies as a workaround — all three were measured against this IP and none defeat an IP-level block. If a call reports the jar as expired, report that and stop: re-exporting it is the operator's job, and `scripts/mcp/youtube-transcript-mcp/README.md` has the procedure.

Full `.vtt` and `.txt` artifacts are cached under `.cache/youtube-transcripts/`; copy research evidence into the active forecast case workspace when it belongs in that case's source pack. Preserve timestamps for quotations. Treat exact figures, product names, and proper nouns from automatic captions as noisy evidence and corroborate them with another source before using them in a forecast.

## Forecasting-system development constitution

These principles govern architecture, method changes and training for both Codex and Claude:

1. **Forecast usefulness is the objective.** A component must improve the generalizable reasoning or auditability of future revenue, operating profit or GAAP attributable net income; test counts, field coverage and scalar scores are diagnostics, never the target.
2. **Correct construction comes before correction.** Put the causal order, accounting basis, evidence permissions and handoffs in the SOP and data contracts. Validators catch residual mistakes; they must not become the main reasoning engine. Checks guard the bottom line (accounting identities, key data actually queried); when checking grows into the system's main mass and drives fix-regenerate loops, that is a design failure to remove, not tune.
3. **Split by capability boundary, not document count or industry.** Create a skill only when it has an independent input/output contract, can be trained and fail independently, and is reusable across companies. Keep orchestration thin and keep sector material optional.
4. **One shared evidence kernel, no copied truth.** Source assets, observations, bounded claims, case-specific evidence use, direct lineage, action logs, evidence requests and immutable snapshots use versioned, storage-neutral contracts shared by every skill. Skills consume/export bundles and never depend directly on SQL, D1 or R2.
5. **Raw and judgment stay separate.** Preserve original bytes and raw values once; normalized values never overwrite raw, derived claims cite direct inputs and rule versions, and case-specific permissions or conflict judgments never mutate the source asset.
6. **Evidence changes forecasts only through explicit reasoning.** LLMs may propose extraction or hypotheses; deterministic transforms handle what they can, material conflicts and model-changing judgments remain reviewable, and newly ingested evidence never silently rewrites a published forecast.
7. **Financial identities are hard; investment judgment is not a score.** Enforce units, periods, segment sums, three-statement rolls and ownership attribution deterministically. Use independent agents and named rivals for causality, source meaning, cycle, moat and persistence; do not collapse these into manual weights or one optimization number.
8. **Tests are orthogonal and retirement-aware.** Each test owns a distinct failure mode. Before adding one, identify overlap and remove or consolidate redundant coverage. Prefer diverse company/regime cases and simpler challengers over fixtures fitted to one issuer.
9. **Snapshots are point-in-time and append-only.** Published forecasts, sealed training cases and their evidence bindings are not edited in place. Corrections create a new version with the changed evidence, rule and reasoning recorded.
10. **Build the smallest system that closes the loop.** Do not add a knowledge graph, vector store, event bus, new top-level object or specialist skill unless a demonstrated forecasting failure requires its independent lifecycle. Complexity must earn its place by changing a material forecast distribution, resolving a rival, or preventing a demonstrated data/accounting error.
11. **Fetch once, reuse through one protocol.** Every acquired source and extracted fact lands in the shared store under stable identity keys (source URI, content hash, accession/period, as-of, retrieved-at) before it is used; a case fetches externally only for gaps the store cannot fill, and sealed/training reads never see material past their cutoff.
12. **Decompose the statements; don't chase indicators.** Start from audited statement lines and their identities, split the lines that matter into computable drivers (product/customer, volume/price, supply/demand, cost blocks) exactly as deep as they can be reasoned cleanly; immaterial residuals follow history or stated experience, material drivers get multi-source challenge, and external papers or expert views cross-check the decomposition instead of becoming a parallel method.
