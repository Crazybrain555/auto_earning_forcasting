# Forecasting skill-system release contract

The live coordinator is `technology-company-profit-forecasting`; the training coordinator is `technology-company-forecasting-trainer`. They share `company-evidence-research`, `company-operating-modeling` and `company-financial-forecasting`, plus the non-skill `forecasting-system-contracts` kernel. Install the five skills side by side from one repository; the method version of any run is the repository commit.

Routing rule: ordinary current-company forecasts, models, and valuations start at the live coordinator, which hands each capability its typed bundle. Historical training rounds, error diagnosis, method revisions, validation and releases start at the trainer; the same specialists run on sealed pre-Actuals bundles. If a specialist receives a request outside its ownership, it returns a typed handoff or evidence request instead of improvising another stage.

Release rule: a revision that passes validation (see `references/historical-training-loop.md`) is released with `scripts/build_skill_system.py --self-test --promote --promotion-evidence <file>`. Specialist and protocol source templates live under the promotion-bound trainer tree; the builder regenerates all production packages and verifies unique stage ownership and exact source/output identity. Promotion accepts only `test_suite.suite_id: trainer_structural_contracts`; the builder resolves and executes that fixed suite itself, never an evidence-supplied command, and judges it by the real return code. Company-specific historical benchmarks remain a separate diagnostic panel. The blind audit binds fixed `assertion_specs` and candidate/challenger `grading_artifacts`; non-compensatory failures are recomputed while causal and generalization judgments remain independent. A failing revision is reverted. Never hand-edit generated coordinators, specialists or contracts, and never commit a partially regenerated system.
